# StackIQ Action Execution (Skill 5 of 5)

The pipeline's terminal stage and its only writer. Every prior skill is read-only; this one changes things — and only ever with the user's explicit approval, within the permission ceiling intake recorded.

## On entry

Read from `state/session.json`: `insight.ordered_finding_ids` (money-descending action priority), the full `diagnostic` block (per-finding `recommendation_hook`, `mode`, and each affected tool's `billing_model`), `intake.access` (the permission ceiling), and `intake.scope` (tools/vendors for execution targeting). Also open `insight.excel_path` — the workbook generated by Skill 4 (Insight) that this skill appends to — and `state/history/run_<n>.json`. Field shapes and the finalized action schema are in the orchestrator's `references/state-schema.md`.

Findings may originate from tools the user named at intake or from Diagnostic's stack-discovery phase — both are already merged into the same `findings[]` / `ordered_finding_ids` before Action ever sees them, so nothing here branches on how a tool entered scope. Only `billing_model` routes the action.

**Re-check open confirmations first.** Before starting new actions, load any `awaiting_confirmation` items from prior history snapshots. For each, check whether the recipient has now confirmed (a reply via the email provider, an explicit user YES this session, or the finding resolved in this run's Diagnostic). If confirmed, flip it to `executed` and update its status row in the workbook. This is how the "wait for confirmation" loop closes across runs.

## Workflow

Process findings **one at a time**, in `ordered_finding_ids` order (money-descending, mirroring the deck). For each finding, run the full cycle below and **persist before moving to the next** — never batch the actions and write the sinks once at the end.

Per finding:

1. **Translate the action by billing model** (never default to seat remediation). Full translation table and routing tree in `references/action-playbook.md` — read it before acting.
2. **Route** by ceiling + boundary: eligible-for-execution, prohibited-boundary (→ manual only), or above-ceiling (→ blocked). See the playbook.
3. **Human approval gate (hard).** No destructive or side-effectful action fires without an explicit user YES. Present the proposed action, its target, its sized impact, and its channel; wait for confirmation. Never infer consent.
4. **Execute** down the channel ladder: MCP → Chrome → manual guidance. When no programmatic path exists, degrade to numbered step-by-step instructions; never silently skip.
5. **Persist immediately — mandatory after every action.** The instant the outcome is known (`executed | awaiting_confirmation | pending_approval | blocked`), record THAT finding before touching the next. Use the bundled helper — do not hand-write the three sinks — so the Excel Action tab, the `session.json` action block, and the `actions` array in `history/run_<n>.json` are updated atomically and keyed by `finding_id` (idempotent upsert, so an `awaiting_confirmation` → `executed` transition updates in place rather than duplicating):

   ```
   python3 "<this skill dir>/scripts/log_action.py" \
     --excel   <insight.excel_path> \
     --session state/session.json \
     --history state/history/run_<n>.json \
     --finding-id <id> --status <executed|awaiting_confirmation|pending_approval|blocked> \
     --action-taken "<what was done>" \
     [--channel mcp|chrome|desktop_app|manual_guidance] [--recipient <email>] \
     [--blocked-reason "<why>"] [--approved-by-user]
   ```

   Requires `openpyxl` (`pip install openpyxl --break-system-packages`). Run it once per action, immediately — never defer to the end of the run. The same call also closes `awaiting_confirmation` items: re-run with `--status executed` when confirmation lands.

After the loop: **reconcile** — confirm every finding in `ordered_finding_ids` has exactly one terminal-status row in the Action tab and in the `action` block (full reconciliation, not just what fired). Fill any missing row before finishing.

## Confirmed action: notify-by-email (the fully-wired action)

StackIQ's one end-to-end confirmed action is a notification email: for a finding, email the responsible party the specific action required (e.g. "revoke these 3 idle Figma seats — ~$45/mo"), then track it to closure. Every other remediation may degrade to manual guidance; this one is wired start to finish. Full steps are in `references/action-playbook.md`; the essentials:

1. **Find a send path — and ask, never assume.** Auto-detect a *send-capable* email connector. Two traps to avoid:
   - **Read/search-only connectors are not a send path.** An Outlook/Microsoft connector that only searches mail cannot send — its absence of a send tool does NOT mean "email won't work."
   - **Never silently send from an off-profile mailbox.** A sender that isn't the user's own internal mailbox — e.g. Apollo's sales mailer — must NOT be auto-selected for an internal note. Treat "only an off-profile or read-only connector exists" the same as "no clean send path": stop and ask.

   When there's no clean same-mailbox send connector, do not block and do not auto-pick. Present the channel menu and let the user choose: **the Outlook desktop app on their computer (via computer use) — prefer this for internal mail**, signed-in webmail via Chrome, another connected email account, a transactional provider (SendGrid/Twilio), an off-profile connector like Apollo *only if the user explicitly okays it*, or having StackIQ draft the email for the user to send. Only after the user confirms none are usable does the action `block` (reason "no email channel available, per user"). The Outlook-desktop steps are in `references/action-playbook.md`.
2. **Ask the recipient at runtime.** Prompt for the recipient address (the admin/tool-owner who can perform the fix). Do not guess it.
3. **Approval gate (hard).** Show the drafted email — recipient, subject, the required action, and the sized impact — and wait for an explicit user YES. No send without it.
4. **Send** via the detected provider.
5. **Set status `awaiting_confirmation`**, not `executed`. Record `recipient` and `sent_date`. Write the status into the workbook Action tab immediately.
6. **Close on confirmation.** The finding flips to `executed` only once the recipient confirms — a reply detected via the email provider, an explicit user YES this session, or the underlying metric resolved on a later Diagnostic run (see the re-check step in On entry).

## Statuses (finalized)

- `executed` — the action ran and was confirmed (MCP response, Chrome DOM check, or a closed confirmation loop).
- `awaiting_confirmation` — the notify-by-email action was approved and sent; waiting for the recipient to confirm or a later run to verify. Carries `recipient` + `sent_date`.
- `pending_approval` — eligible and resolvable by a user YES, but not yet approved.
- `blocked` — not resolvable by approval alone: above the `intake.access` ceiling, on the prohibited-action boundary, or no path at all. Do not block the email action just because no send-capable connector auto-detected — first ask the user for a channel (see step 1). Always carries a `blocked_reason`.

These are distinct; do not conflate `awaiting_confirmation` (sent, unconfirmed) with `pending_approval` (unsent, unapproved) or `blocked` (unresolvable).

## The three sinks (all dated, finalized schema)

1. **`action` block in `session.json`** — append:
   ```json
   "action": {
     "executed":              [ { "finding_id": "", "action_taken": "", "channel": "", "date": "" } ],
     "awaiting_confirmation": [ { "finding_id": "", "action_taken": "", "channel": "", "recipient": "", "sent_date": "" } ],
     "pending_approval":      [ { "finding_id": "", "action_taken": "", "date": "" } ],
     "blocked":               [ { "finding_id": "", "action_taken": "", "blocked_reason": "", "date": "" } ]
   }
   ```
2. **Action tab on the Insight workbook** (`insight.excel_path`, via the `xlsx` skill) — append a clearly-marked new tab; NEVER mutate Skill 4's finding rows. One row per finding, ordered by `ordered_finding_ids`, columns: `finding_id`, `action_taken`, `status`, `blocked_reason`, `channel`, `recipient`, `approved_by_user`, `sent_date`, `confirmed_date`, `date`. Keyed to `finding_id` for reconciliation. The status column is updated in place as a finding moves `awaiting_confirmation` → `executed` (still append-only against Skill 4's finding rows — only the Action tab's own status cell changes).
3. **`actions` array in `state/history/run_<n>.json`** (immutable) — append `{ finding_id, action_taken, status, channel, recipient, approved_by_user, sent_date, confirmed_date, date }`. This is the cross-run continuity source next week's Diagnostic reads to avoid resurfacing resolved findings, to carry open ones forward, and to close `awaiting_confirmation` items once verified.

On completion set `stage.status.action = done` and `stage.current = action` — the run is closed.

## Guardrails

- **Persist after every action, never batch.** The Excel Action tab and all three sinks are updated the instant each single action resolves, one finding at a time, before the next finding is touched. Deferring writes to the end of the run is the top cause of inconsistent/ missing rows — do not do it.
- **Approval gate is hard** — nothing destructive or side-effectful executes without an explicit user YES.
- **Respect the `intake.access` ceiling** — never attempt an action above the recorded permission level.
- **Prohibited-action boundary stays human-only** regardless of approval: permission/access changes, billing/subscription changes, and hard deletes are never auto-executed — they go straight to manual guidance.
- **`blocked` ≠ `pending_approval`.**
- **Append-only** — the workbook Action tab and the history actions array are appended, keyed by `finding_id`; finding rows are immutable.
- **Never silently skip** — a non-programmatic action degrades to manual guidance, and every finding gets a terminal-status row.
- **Billing-model-aware** across SaaS / AI / API products (seat / consumption / flat / tiered) — regardless of whether the tool was named at intake or surfaced by stack discovery.

## Handoff

This closes the pipeline. The `action` block returns control to the orchestrator (run complete); the `actions` array in `history/run_<n>.json` feeds next week's Diagnostic for week-over-week "was this actioned, and when." No downstream skill — this is the terminal stage.
